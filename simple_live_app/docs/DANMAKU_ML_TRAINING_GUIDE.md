## 1. 目标概述

### 1.1 最终目标

```
弹幕监控 → ML预处理 → AI总结 → 时间线输出
```

通过训练一个本地小模型，对弹幕进行实时质量分类，结合 ASR 字幕，最终交给 AI 生成直播内容摘要。

### 1.2 设计决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| **用户信息** | 不带用户ID和UID | 保护隐私，减少数据量 |
| **模型输入** | 仅文本内容 + 时间戳 | 简化处理 |
| **表情包** | 解析含义，用于场景判断 | 打call多=唱歌，?=疑惑 |
| **SC** | 权重最高，单独处理 | 付费内容，价值密度大 |
| **ASR字幕** | 仅在 speech 时段使用 | 音乐时段可能产生错误识别 |

### 1.3 技术指标

| 指标 | 目标值 |
|------|--------|
| 推理延迟 | < 10ms/条（CPU） |
| 模型大小 | < 100MB |
| 准确率 | > 85% |
| 训练时间 | < 24 小时（GPU） |

### 1.5 ASR 字幕过滤

由于 ASR 模型（sherpa_onnx）仅支持中文，当主播播放外语歌曲时，ASR 会产生大量错误文本。使用 `inaSpeechSegmenter` 实时检测音频类型：

| 音频状态 | 说明 | ASR 处理 |
|---------|------|---------|
| `speech` | 主播在说话 | 使用 ASR 字幕 |
| `music` | 播放音乐/唱歌 | 丢弃 ASR 字幕 |
| `noEnergy` | 静音 | 等待 |

参考项目：`D:\AHEU\code\bili-music-segmenter`

## 3. 数据清洗

### 3.1 清洗目标

1. 提取纯文本弹幕内容（保留有含义的表情包，解析其含义）
2. 去除空白/无效弹幕
3. 统一编码为 UTF-8
4. 不保留用户ID和UID

### 3.2 内容类型与权重

| 类型 | 权重 | 说明 |
|------|------|------|
| **SC（醒目留言）** | 最高 | 付费内容，用户投入成本高，内容价值密度大 |
| **弹幕文本** | 中 | 普通弹幕，过滤后保留有价值的 |
| **表情包** | 低 | 统计为主，不进入AI文本输入 |

### 3.3 数据结构

**有价值弹幕**
```json
{
  "room_id": "22642754",
  "timestamp": "1771228478444",
  "content": "主播讲讲芯片吧",
  "type": "danmaku",
  "label": "valuable"
}
```

**SC（醒目留言）**
```json
{
  "room_id": "22642754",
  "timestamp": "1771228478440",
  "content": "主播过年期间会播游戏吗？",
  "type": "superchat",
  "price": 6.0,
  "label": "important"
}
```

**表情包**
```json
{
  "room_id": "22642754",
  "timestamp": "1771228478500",
  "content": "打call",
  "type": "emoji",
  "emoji_meaning": "singing"
}
```

### 3.4 表情包含义映射

| 表情包内容 | 含义 | 场景标签 |
|-----------|------|---------|
| 打call | 支持/应援 | singing |
| 棒/好棒 | 正面反馈 | positive |
| ？/？？？ | 疑惑/争议 | confusing |
| 666 | 厉害 | impressive |
| 哈哈哈 | 好笑 | funny |

### 3.5 标签说明

| 字段 | 可能的值 | 说明 |
|------|---------|------|
| `type` | danmaku, superchat, emoji | 内容类型 |
| `label` | valuable, normal, spam | 质量标签 |
| `emoji_meaning` | singing, positive, confusing, impressive, funny | 表情含义 |

## 4. 数据标注

### 4.1 标注策略

由于数据量庞大（百万级），采用**规则标注 + 人工抽检**的策略。

### 4.2 标注规则

| 规则 | 内容 | 标签 |
|------|------|------|
| **SC** | 所有醒目留言 | 1（有价值） |
| 刷屏模式 | 哈哈哈、66666、23333、纯字母/数字 | 0（无价值） |
| 过短 | 长度 ≤ 2 且无含义 | 0（无价值） |
| 纯表情包 | 不可识别含义的表情包 | 0（无价值） |
| 问句 | 包含？/吗/呢/嘛 | 1（有价值） |
| 讨论内容 | 包含名词+动词，有实际语义 | 1（有价值） |
| 有含义表情包 | 打call、棒、好棒 | 1（有价值） |

### 4.3 人工抽检

运行自动标注后，抽取 50-100 条样本人工检查，判断准确率是否 > 80%。

## 5. 模型选择与训练

### 5.1 模型选择

使用中文预训练模型进行微调：

| 模型 | 参数量 | 模型大小 | 推荐度 |
|------|--------|---------|--------|
| `hfl/chinese-bert-wwm-ext` | 102M | ~400MB | ⭐⭐⭐⭐⭐ |
| `hfl/chinese-roberta-wwm-ext` | 102M | ~400MB | ⭐⭐⭐⭐⭐ |
| `distilbert-base-chinese` | 66M | ~250MB | ⭐⭐⭐⭐ |

推荐使用 `hfl/chinese-bert-wwm-ext` 或 `hfl/chinese-roberta-wwm-ext`。

### 5.2 数据划分

| 数据集 | 比例 | 说明 |
|--------|------|------|
| 训练集 | 80% | 用于训练 |
| 验证集 | 10% | 调整超参数 |
| 测试集 | 10% | 最终评估 |

### 5.3 训练环境

- **Python 环境**：conda 或 venv
- **深度学习框架**：PyTorch + Transformers
- **硬件**：GPU（推荐 NVIDIA GTX 1060 及以上）
- **无 GPU**：可使用 Google Colab（免费 T4 GPU）

### 5.4 训练步骤

1. 安装依赖：`pip install torch transformers datasets pandas scikit-learn`
2. 加载数据，划分训练/验证/测试集
3. 加载预训练 tokenizer
4. 创建数据集（使用 PyTorch Dataset）
5. 配置训练参数（学习率、批次大小、训练轮数）
6. 开始训练（使用 HuggingFace Trainer）
7. 在测试集上评估
8. 保存模型和 tokenizer

### 5.5 训练输出

```
output/danmaku_classifier/final_model/
├── config.json              # 模型配置
├── pytorch_model.bin     # 模型权重
├── tokenizer_config.json  # tokenizer 配置
├── vocab.txt              # 词表
└── training_report.json  # 训练报告
```

## 6. 模型导出与集成

### 6.1 导出为 ONNX

使用 PyTorch 的 `torch.onnx.export` 将模型导出为 ONNX 格式，便于在 Flutter 中使用。

### 6.2 Flutter 集成

使用 `onnxruntime_flutter` 加载模型并进行实时推理。推理延迟预计：**< 10ms/条（CPU）**

### 6.3 预处理流程

```
弹幕输入
    │
    ├── 模型分类判断是否有价值
    │
    ├── 有价值 → 保留文本内容 + 时间戳
    │
    └── 表情包 → 统计类型和数量 → 场景状态判断
```

## 7. 直播内容总结流程

### 7.1 分割策略

| 策略 | 说明 |
|------|------|
| **时间窗口** | 每 N 分钟切一刀，作为兜底 |
| **弹幕密度** | 连续 X 秒弹幕 < Y 条时提前切割（说明话题间隙） |
| **场景切换** | 检测到 scene_tags 变化时切割 |
| **SC出现** | 有SC时可以优先作为新话题的起点 |
| **音乐检测** | 检测到 music 时可作为分割点 |

```
│  1. 接收数据 → POST /api/live/summary/push_batch              │
│  2. 认证 + 限流 → 检查 access_token、推送频率                   │
│  3. 数据校验 → 验证字段格式、过滤无效数据                      │
│  4. 缓存 + 聚合 → 按 room_id 缓存当前时间窗口                  │
│  5. AI 总结生成 → 构造 Prompt、调用 AI API、解析返回          │
│  6. 存储 + 推送 → 写入数据库、WebSocket 推送                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 9. 常见问题

### Q1: 训练需要 GPU 吗？

是的，强烈建议使用 GPU。CPU 训练会非常慢（可能需要几天）。

**免费 GPU 选项**：
- Google Colab（免费 T4 GPU）
- Kaggle Notebooks（免费 P100 GPU）

### Q2: 模型太大了怎么办？

选择更小的预训练模型：`distilbert-base-chinese`（66M 参数，~250MB）

### Q3: 实时推理太慢怎么办？

- 减少文本最大长度（MAX_LEN 从 64 减到 32）
- 使用更小的模型
- 使用批量推理（一次处理多条）
